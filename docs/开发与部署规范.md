# 开发与部署规范

## 模块边界

- `frontend/`：页面、样式和素材；不直接访问数据库。
- `backend/api/`、`backend/auth/`、`backend/admin*.py`：HTTP 接口层；业务 API 按领域放在 `backend/api/routers/`。
- `backend/storage.py`：唯一图片和前端素材存储入口。
- `backend/services/data_service.py`：统一数据库删除、OSS 清理和派生计数。
- `backend/services/`：业务流程、外部 AI 和监控服务。
- `backend/db/`：模型、数据库连接和 Alembic 迁移。
- `scripts/`：启动、迁移、部署入口；不要把密钥写入脚本。

## 数据库变更

1. 先改 `backend/db/models.py`。
2. 生成一条新的 Alembic migration，不修改已发布的历史 migration。
3. 本地执行 `bash scripts/migrate_db.sh`。
4. 通过测试后再提交；ECS 部署脚本会先构建镜像，再执行 `alembic upgrade head`，最后启动服务。

数据库文件和 Docker volume 不随代码更新删除，迁移版本以 `alembic_version` 为准。

## 小步迭代

每次提交只做一个主题：

- `ui`：页面、样式、素材路径。
- `api`：请求/响应契约。
- `service`：业务逻辑和外部服务。
- `db`：模型与迁移。
- `ops`：Docker、部署和监控。

跨模块需求拆成连续的小提交，并在提交信息中标明主题，例如：
`feat(ui): add postcard detail`、`fix(oss): normalize asset url`、`db: add postcard index`。

## 部署原则

- ECS 使用 `docker-compose.prod.yml`，镜像包含代码和前端素材，不挂载源码、不启用 `--reload`。
- 本地开发使用 `docker-compose.yml`，允许源码挂载和热重载。
- `.env`、数据库 volume、生成图片 volume 永远不由部署脚本覆盖或删除。
- 部署完成必须通过 `/api/admin/health`，失败时保留容器日志用于回滚排查。
